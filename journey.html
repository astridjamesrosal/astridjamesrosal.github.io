Bank System

Heading next to the Bank System, this project handles multiple classes now which are Transaction, Account, and Bank. These classes depend on values from one another to complete an action. For example, Bank.deposit() calls get_account first before proceeding with said action, and then the action method constructs a Transaction that contains all the details that happened. It is a collaboration instead of just the classess coexisting.

Also, StudentRegistry.add checks for the uniqueness of a student, if they already exist, its not valid, but if not, it allows the creation of one. Persistence is also demonstrated here via FileManager for it saves all the data of the students, their name, ID number, year level, and grade.

Library System

For the last project in Python OOP, the Library system introduces status transition validation before saving so it only executes on real change. We first check if the book is_available is True, and depending on the method if it is borrow or return, it would change the status of the book of is_available to False, making users that try to borrow it would be unsuccessful if the book is already borrowed. Then if the action is return, and the status of the book is Is_available is False, after the method, it would now be set to True, and then it would be saved. Therefore, when a user tried to borrow the book, it would now be allowed.

Pando - Study Session Tracker

After the four Object Oriented Programing Projects, we move on to Flask. In the first project, Pando, A Study Session Tracker, I was introduced to how Flask communicates with webpages through routes like `add_session_route` and `delete_session_route`, which call into database functions such as `add_session` and `delete_session`, but no edit routes at all, so once a session is created, you simply can view or delete it. I also learned Jinja syntax which taught me how to be efficient with rendering conditionals and loops in templates. Input validation lives entirely in the Flask route, not the database and the schema itself has no constraints such as no NOT NULL. And lastly, Sessions is a single table with no foreign keys which means there is no other entity for it to relate to.

Orchid - Finance Tracker

And finally, the last project before college, Orchid, a Finance Tracker, builds on top of app-level routing by introducing relational data with foreign keys and JOINs. Because Transactions reference Accounts and Categories, hard-deleting either would break linked transactions which is why Accounts and Categories use soft delete instead. For transactions, once deleted, it is hard-deleted and it leaves no audit trail. NOT NULL constraints also enforce that every field has input, a requirement Pando's schema never had. Unlike Pando, Orchid has editability for Accounts, Categories, and Transactions which would be very useful for users if they ever make a valid input that is factually wrong. Through this feature, users can correct a mistaken entry rather than deleting and creating one again. JOINs connects the three tables, Accounts, Categories, and Transactions for functions like get_all_transactions because it needs the data from those three tables to produce the complete details of a transaction.
